## 1. 제품 개요

### 1.1 배경

**왜 이 제품이 필요한가?**

공공기관, 사회복지시설, 교육기관 등에서 정부문서화일(황대일 파일) 라벨을 출력하는 업무는 **주기적으로 반복되는 단순 노동**입니다. 현재 대부분의 실무자는 인터넷에서 다운로드한 HWP 양식 파일에 텍스트를 직접 입력하고, 글자가 넘치면 장평/자간을 수동으로 조절하는 '노가다'를 반복하고 있습니다.

| 현재 문제 | 영향 |
|----------|------|
| 장평/자간 수동 조절 | 건당 5~15분 추가 소요 |
| 엑셀 → HWP 복붙 반복 | 100건 = 100번 복붙 |
| 더블체크 필요 | 불안감 + 재출력 리스크 |
| 설치형 프로그램 사용 불가 | 공공기관 보안 정책으로 설치 제한 |

**현재 시장 상황은?**

- **직접 경쟁:** 폼텍 디자인 프로 9, 프린텍 라벨메이커 (모두 설치 필수)
- **간접 경쟁:** 한글 HWP 양식 파일 (무료, 익숙함)
- **공백 영역:** "정부문서화일 전용 + 웹 기반 + 자동 텍스트 피팅" 조합 서비스 **없음**

### 1.3 핵심 가치 제안 (UVP)

> **"정부문서화일 라벨을 직접 뽑아야 하는 실무자"는  
> "긴 부서명을 넣으면 글자가 넘쳐서 장평 줄이는 노가다"를 할 때  
> "라벨 메이커"를 통해  
> "입력만 하면 자동으로 예쁘게 맞춰진 PDF"를 얻는다**

**킬러 피처:** 자동 텍스트 피팅 (Smart Fitting)
- "대전광역시아동보호전문기관"을 입력해도 알아서 장평/자간 조절
- 사용자는 크기 걱정 없이 텍스트만 입력

## 3. 기능 정의

### 3.1 기능 우선순위

#### 🔴 Must Have (MVP 필수)

| 기능 | 설명 | 사용자 스토리 |
|------|------|--------------|
| **단일 라벨 입력 폼** | 제목, 생산연도, 부서명, 분류번호, 보존기간, 관리번호 입력 | 사용자로서 필요한 정보를 입력하면 라벨이 만들어지길 원한다 |
| **자동 텍스트 피팅** | 긴 텍스트 입력 시 자동으로 폰트 크기/장평 조절 | 사용자로서 글자가 삐져나가지 않게 자동 조절되길 원한다 |
| **실시간 미리보기** | 입력과 동시에 라벨 미리보기 렌더링 | 사용자로서 출력 전에 어떻게 나올지 확인하고 싶다 |
| **PDF 다운로드** | A4 사이즈 PDF로 다운로드 (폼텍 3629 규격) | 사용자로서 바로 출력할 수 있는 파일을 받고 싶다 |
| **3종 라벨 동시 출력** | 표지용 + 측면용 + 내부 우상단용 라벨 한 번에 생성 | 사용자로서 세 종류 라벨을 따로 만들지 않고 싶다 |

#### 🟡 Should Have (출시 후 2주 내)

| 기능 | 설명 |
|------|------|
| **입력 기록 저장** | 브라우저 LocalStorage에 최근 입력값 저장 |
| **복사하기 기능** | 이전 라벨 복사해서 일부만 수정 |
| **사용 가이드** | 첫 방문자용 간단한 사용법 안내 |

#### 🟢 Nice to Have (추후 검토)

| 기능 | 설명 |
|------|------|
| **엑셀 대량 변환** | 엑셀 업로드 → 다중 라벨 PDF 생성 |
| **기관 템플릿 저장** | 자주 쓰는 부서명/분류번호 저장 |
| **회원 가입** | 클라우드에 라벨 기록 저장 |

#### 🔵 Won't Have (이번엔 안 함)

| 기능 | 안 하는 이유 |
|------|-------------|
| 모바일 앱 | 웹으로 충분, 개발 리소스 집중 |
| 바코드/QR 코드 | 정부문서화일에는 불필요 |
| 다국어 지원 | 한국 시장 전용 서비스 |
| 유료 결제 시스템 | MVP 단계에서 불필요, 후원만 받음 |

### 3.2 핵심 사용자 플로우

```
┌─────────────────────────────────────────────────────────────┐
│                      메인 페이지                              │
├─────────────────────────────────────────────────────────────┤
│  ┌─────────────────────┐    ┌─────────────────────────────┐ │
│  │     입력 폼          │    │       실시간 미리보기        │ │
│  │                     │    │                             │ │
│  │  제목: [_________]  │    │   ┌─────────────────────┐   │ │
│  │  연도: [_________]  │ →  │   │    표지 라벨 미리보기  │   │ │
│  │  부서: [_________]  │    │   ├─────────────────────┤   │ │
│  │  분류: [_________]  │    │   │    측면 라벨 미리보기  │   │ │
│  │  보존: [_________]  │    │   ├─────────────────────┤   │ │
│  │  관리: [_________]  │    │   │   내부 라벨 미리보기   │   │ │
│  │                     │    │   └─────────────────────┘   │ │
│  │  [PDF 다운로드 버튼] │    │                             │ │
│  └─────────────────────┘    └─────────────────────────────┘ │
└─────────────────────────────────────────────────────────────┘
```

**상세 플로우:**

1. **사용자가** 메인 페이지에 접속한다
2. **사용자가** 입력 폼에 라벨 정보를 입력한다 (제목, 연도 필수 / 나머지 선택)
3. **시스템이** 입력과 동시에 우측 미리보기를 실시간 렌더링한다
4. **시스템이** 긴 텍스트는 자동으로 폰트 크기를 줄여 피팅한다
5. **사용자가** 미리보기를 확인하고 "PDF 다운로드" 버튼을 클릭한다
6. **시스템이** 폼텍 3629 규격에 맞는 PDF를 생성한다
7. **결과:** 사용자는 바로 출력 가능한 PDF 파일을 다운로드 받는다

## 4. 데이터 모델 (Supabase 테이블)

### document_labels 테이블

| 컬럼명 | 데이터 타입 | Key / 설정 | 설명 |
|--------|------------|------------|------|
| **id** | `int8` | PK, Is Identity | 고유 ID (자동 증가) |
| **created_at** | `timestamptz` | Default: `now()` | 데이터 생성 시간 |
| **batch_id** | `uuid` | Index | 업로드 그룹 ID (비회원 식별용) |
| **title** | `text` | Not Null | 제목 (필수) |
| **production_year** | `text` | Not Null | 생산연도 (문자열로 유연하게) |
| **department_name** | `text` | Nullable | 부서명 |
| **classification_code** | `text` | Nullable | 분류번호 |
| **retention_period** | `text` | Nullable | 보존기간 |
| **management_number** | `text` | Nullable | 관리번호 |

### 설계 참고사항

> [!IMPORTANT]
> **`batch_id` 설계 이유:**
> - 로그인 없이 비회원 사용 지원
> - 프론트엔드에서 UUID 생성 → `batch_id`에 저장
> - 브라우저 LocalStorage에 `batch_id` 저장 → "내 라벨 다시 보기" 기능 구현

> [!NOTE]
> **RLS 정책 (MVP 단계):**
> - `Enable RLS` 체크 해제 또는
> - Policy: `Allow public access`, Target: `anon`, CHECK: `true`

> [!TIP]
> **`production_year`를 text로 한 이유:**
> - "2024~2025", "2025(상반기)" 등 유연한 입력 허용
> - int로 하면 예외 케이스에서 에러 발생

## 8. PRD 약점 분석

### 가장 큰 약점 1가지

> **"실제 사용자 검증 없이 기획된 PRD"**

현재 PRD는 PM의 가정과 시장 조사에 기반하고 있으며, 실제 타겟 사용자(사회복지사, 공무원)의 인터뷰나 프로토타입 테스트를 거치지 않았습니다.

**발생 가능한 문제:**
- 가정한 페인포인트가 실제로는 크지 않을 수 있음
- 기대한 사용 흐름과 실제 사용 흐름이 다를 수 있음
- "자동 피팅"보다 다른 기능이 더 필요할 수 있음

### 보완 방법

| 단계 | 액션 | 타이밍 |
|------|------|--------|
| **1. 빠른 프로토타입** | Figma로 클릭 가능한 프로토타입 제작 | 개발 전 1주 |
| **2. 타겟 인터뷰** | 사회복지사 3명 + 공무원 2명 인터뷰 | 개발 전 1주 |
| **3. 랜딩 페이지 테스트** | 서비스 설명 랜딩만 먼저 오픈 → 관심도 측정 | 개발 1주차 |
| **4. 베타 테스트** | MVP 완성 후 5~10명 실사용 테스트 | 개발 3주차 |

> [!CAUTION]
> 사용자 검증 없이 4주 개발 후 "아무도 안 쓰는" 상황을 피하려면, **1주차에 최소한 타겟 인터뷰 3건**은 필수입니다.
